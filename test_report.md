## 2. 세부 검증 내용 및 진단 결과

### 가. 모듈별 API 서버 헬스체크 (test_module_health_checks)
* 검증 방식: 각 모듈의 API 문서 주소(/docs)로 실제 HTTP GET 요청을 전송하여 정상 응답(200 OK) 수신 여부를 확인합니다.
* 진단 상세:
  * 004 Database API (포트 8004): 연결 성공 (정상 가동 중)
  * 014 Orchestrator API (포트 8014): 연결 성공 (정상 가동 중)
  * 011 Processability API (포트 8011): 연결 실패 (Connection Error - 서버 오프라인)
  * 012 Matching API (포트 8012): 연결 실패 (Connection Error - 서버 오프라인)
  * 013 Reverse Engineering API (포트 8013): 연결 실패 (Connection Error - 서버 오프라인)
* 진단 요약: 004 DB 및 014 오케스트레이터는 정상적으로 요청을 수신하고 있으나, 연동 대상인 011, 012, 013 서비스가 오프라인 상태임을 정확히 감지하여 에러 보고를 수행했습니다.

### 나. E2E 통합 워크플로우 테스트 (test_full_pipeline_e2e)
* 검증 방식: 004 데이터베이스 모듈에서 실제 피착재 정보를 쿼리하여 오케스트레이터의 입력값으로 가공한 후, 최종 파이프라인 API를 구동합니다.
* 테스트 흐름:
  1. 004 DB 조회: roughness_md_max=1.0 기준으로 피착재 정보를 탐색하여 아래의 실데이터를 정상적으로 획득했습니다.
     * 획득 데이터 예시: ID 1, 제조사 대스틸, 제품명 BA, 조도 MD 0.02, 표면에너지 MD 38.0
  2. 오케스트레이터 요청: 획득한 피착재 정보(조도, 표면에너지 등)를 바탕으로 014 오케스트레이터의 /orchestrate 엔드포인트에 통합 요청을 전송했습니다.
  3. 예외 및 Fallback 제어 흐름 검증: 011, 012, 013 모듈이 오프라인인 환경 하에서도 오케스트레이터 내부의 예외 처리 프로토콜이 정상적으로 동작하여, 실패하지 않고 기본 Fallback 결과(status: 'reverse_engineered')가 200 OK 응답과 함께 전달됨을 확인했습니다.
